# Cloudflare D1 병목 현상 원인 분석 및 해결 방안

**[진행율]** ✅✅✅✅✅ 100%, 예상 잔여시간 ⌛ 0분 (작업 완료)

---

## 1. Planning (계획) & Analysis (분석)
**[본질가]**
문제의 본질은 서버리스 환경에서 Cloudflare D1(SQLite 기반)의 동시성 한계와 트랜잭션 크기 제한에 있소. 여러 크롤링 워커나 트리거가 동시에 대량의 INSERT(ON CONFLICT DO UPDATE) 쿼리를 시도하면 `SQLITE_BUSY`(DB 락)가 발생하여 데이터 적재가 누락되는 것이오. 또한 `body_html` 등 무거운 페이로드를 단일 Batch로 밀어넣으면 D1 요청 크기 제한 및 쓰기 병목에 부딪히오.

**[회의론자]**
아이고, 배치 사이즈 30에 딜레이 없이 밀어붙였으니 당연히 터지지 않겠습니까? 서버리스 워커는 CPU 시간 50ms, 전체 실행 시간 10~30초 타임아웃도 엄격합니다. 무턱대고 딜레이만 주면 이번엔 타임아웃으로 프로세스 자체가 죽어버릴 겁니다. 적재 실패를 근본적으로 막으려면 비동기 메시지 큐(Queue)가 필수입니다.

---

## 2. Design (설계)
**[사용자 대변인]**
원인이 그렇다면 당장 서버가 뻗지 않게 응급 처치부터 해주세요 ㅠㅠ! 복잡한 인프라 공사(Queue 등)는 당장 하기 힘드니, 현재 소스코드 베이스에서 바로 적용할 수 있는 쉬운 코드 수정안을 부탁드려요.

**[통합자]**
여러분의 의견을 종합하여 투트랙(Two-track) 방안을 설계합니다.
1. **응급 처치 (단기적해결)**: `server/src/index.ts` 코드 내 `DB_BATCH_SIZE`를 10으로 하향 조정하고, D1이 쓰기 락을 해제할 시간적 여유를 가질 수 있도록 청크 삽입 간 100~150ms의 딜레이를 부여합니다.
2. **아키텍처 개선 (장기적해결)**: Cloudflare Queues를 파이프라인에 추가합니다. 크롤러는 Queue에 데이터를 쏘고 종료되며, 별도의 Consumer 워커가 순차적으로 단일 트랜잭션으로 D1에 데이터를 적재하여 병목을 원천 차단합니다.

## 3. Implementation (구현 방안) & Inspection (검수)
**[실천가]**
말 길어. 당장 코드 안 뻗게 수정할 부분은 이거임. `index.ts` 열어서 바로 반영해.

1. DB_BATCH_SIZE 축소 (30 -> 10)
```typescript
const DB_BATCH_SIZE = 10; 
```

2. 배치 청크 간 딜레이 삽입 (락 릴리스 유도)
```typescript
async function insertPostsBatch(db: D1Database, posts: Post[]) {
  if (posts.length === 0) return;
  // ... 생략 ...
  for (let i = 0; i < posts.length; i += DB_BATCH_SIZE) {
    const chunk = posts.slice(i, i + DB_BATCH_SIZE);
    
    // ... DB INSERT 처리 ...
    await db.batch(statements);

    // 타임아웃에 안 걸릴 정도의 짧은 대기 시간 (D1 쓰기 락 해소 유도)
    await new Promise(resolve => setTimeout(resolve, 150));
  }
}
```

3. (장기 개선용) Queues 인프라 도입 (`wrangler.toml` 추가 예시)
```toml
# 데이터를 던지는 Producer 워커 바인딩
[[queues.producers]]
queue = "community-aggregator-queue"
binding = "POST_QUEUE"

# 큐 데이터를 차례대로 D1에 적재하는 Consumer 워커 설정
[[queues.consumers]]
queue = "community-aggregator-queue"
max_batch_size = 10
max_batch_timeout = 5
```

**[회의론자]**
150ms 딜레이와 사이즈 조율은 임시방편으로는 나쁘지 않습니다만, 결국 일일 크롤링 한도와 워커 CPU 타임을 주기적으로 모니터링해서 Queue 구조로 빨리 넘어가는 게 보안관례상 안전합니다.

**[통합자]**
검수가 모두 완료되었습니다. 위 개선 방안을 코드로 반영하면 D1 병목으로 인한 잦은 실패 빈도를 획기적으로 낮출 수 있습니다.
