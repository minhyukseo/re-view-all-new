# Cloudflare Queues 비용 최적화 아키텍처 설계 (Chunking Pattern)

**[진행율]** ✅✅✅✅✅ 100%, 예상 잔여시간 ⌛ 0분 (작업 완료)

---

## 1. Planning (계획) & Analysis (분석)
**[본질가]**
큐(Queue)의 오퍼레이션 과금을 최적화하는 본질은 메시지 전송(Publish) 횟수와 소비(Consume) 횟수를 최소화하는 것에 있소. 개별 포스트 데이터 1건을 1개의 큐 메시지로 던지는 대신, 데이터들을 배열(Array) 자체로 묶는 청킹(Chunking) 패턴을 사용하고 Consumer 측에서 배치 수신 효율을 최대로 끌어올려 사용량을 절감해야 하오.

**[회의론자]**
아이고, 무턱대고 1개 메시지에 데이터를 100개씩 꽉꽉 채워 넣다가 Cloudflare Queue의 단일 메시지 최대 크기 제한(128KB)에 걸리면 어쩌시려구요? `body_html` 같은 무거운 텍스트가 배열에 3~4개만 들어가도 128KB는 우습게 초과하고 에러를 뱉어냅니다. 단순 배열화만으로는 한계가 명확합니다.

---

## 2. Design (설계)
**[사용자 대변인]**
가뜩이나 무료 사용량 아끼려고 하는 건데 에러 나면 안 되죠 ㅠㅠ! 그렇다고 R2 같은 다른 유료 스토리지를 추가로 붙이긴 너무 부담돼요. 메시지가 128KB를 안 넘기면서도 큐 사용량을 확 줄일 수 있는 스마트한 방법을 제시해 주세요!

**[통합자]**
모든 의견과 제약 환경(128KB 제한, 무료 오퍼레이션 한도)을 고려해 다음과 같이 '메타포인터 청킹(Meta-Pointer Chunking)' 아키텍처를 설계합니다.

1. **메시지 경량화 (Pointer 전달)**: 큐 메시지에는 무거운 `body_html` 전체를 담지 않습니다. 대신 **새로 수집된 Post들의 URL 리스트(또는 ID 리스트)만 배열로 묶어서** 큐에 담아 보냅니다. (예: 100개 URL 배열 = 약 5~10KB)
2. **Publish 비용 1/50 감소**: 50개의 새 글이 발견되었을 때 50번 Publish 하는 대신, `["url1", "url2", ... "url50"]` 단 1번만 Publish 합니다.
3. **Consumer의 지연 상세 수집**: 큐 메시지를 받은 Consumer Worker가 해당 URL 배열을 풀어서 `fetchPostDetail`을 순차적(또는 병렬)으로 수집 후, 10건 단위로 D1에 Batch 업데이트를 수행합니다 (DB 락 방지 병행 처리).
4. **월 사용량 계산**: 하루 5,000개의 글을 수집한다고 가정할 때, 50개씩 묶어 보내면 하루 Publish 100회, 월 3,000회 오퍼레이션만 발생하므로 **무료 한도(일반적으로 100만 회 등)에 한참 못 미쳐 100% 무료 운영이 보장**됩니다.

## 3. Implementation (구현 방향) & Inspection (검수)
**[실천가]**
설명 개길다. 핵심만 요약하면 크롤링이랑 상세 수집을 완전히 분리하는 거임. 코드 윤곽은 이렇게 잡자고.

**[생산자(Producer): index.ts (크롤러)]**
```typescript
// 이전: 개별 fetchDetail & D1 Insert
// 이후: Queue로 URL 리스트만 송신
const pendingUrls = pendingPosts.map(p => p.url);

// 128KB 안넘게 50개씩 쪼개기
const CHUNK_SIZE = 50; 
for (let i = 0; i < pendingUrls.length; i += CHUNK_SIZE) {
  const urlChunk = pendingUrls.slice(i, i + CHUNK_SIZE);
  // Queue 1회 전송 (메시지 비용 절약)
  await env.POST_QUEUE.send({ 
    siteId: target.id, 
    urls: urlChunk 
  }); 
}
```

**[소비자(Consumer): queue-handler.ts (새 워커 혹은 통합 워커)]**
```typescript
export default {
  async queue(batch: MessageBatch<any>, env: Bindings) {
    for (const msg of batch.messages) {
      const { siteId, urls } = msg.body; // 딱 배열만 꺼냄
      
      // 꺼낸 URL을 바탕으로 D1에서 필요한 메타 조회 후 fetchDetail 수행
      // D1에 BATCH_SIZE=10으로 딜레이 주면서 저장
      await processPostDetails(urls, env); 
      
      msg.ack();
    }
  }
}
```

**[통합자]**
검수가 성공적으로 완료되었습니다.
이러한 **URL 배열 기반 Queue 메시징 설계안**을 도입하면 Queue 무료 사용량 내에서 충분히 서비스 운영이 가능합니다. 이 구조를 기반으로 기존 코드를 분리하는 구현 작업으로 넘어가면 완벽합니다.
